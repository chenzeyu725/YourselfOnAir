# 产品规划文档（V1 讨论稿）

> 项目代号：**YourselfOnAir**  
> 日期：2026-04-11（UTC）  
> 定位：面向个人/团队的“工作习惯蒸馏 + 专家蒸馏 + 企业知识问答”订阅式 Web SaaS。

---

## 0. 你给出的目标（我先统一成可执行定义）

你希望做一个网页端订阅系统（可用 `@build-web-apps` 插件快速落地），具备：

1. 上传并管理多模态知识数据（聊天记录、文档、图片、日志等）。
2. 学习用户工作习惯，回答工作问题，提供工作思路。
3. 可生成办公产物：PPT、简单 Excel、Word、PDF。
4. 部署到你自己的服务器。
5. 用户可检索/下载所有自己上传文件，并可配置“特定对外口径”。
6. 两个产品线：
   - **蒸馏自己**（参考 yourself / colleague / pig）
   - **蒸馏 Expert**（参考 nuwa）

## 1. 参考项目拆解（对你产品有价值的“能力原子”）

### 1.1 yourself-skill / colleague-skill / pig-skill 的共性价值

这类项目共同指向一件事：**从真实历史数据中提取可复用的人格与工作行为模式**，并将其封装为可调用的 AI 角色。

可借鉴能力：
- 多来源数据接入（聊天、文档、截图、图片等）。
- 将数据拆成两层：
  - 记忆层（事实、经历、上下文）
  - 人格层（语气、习惯、偏好、决策方式）
- 以“像某个人那样回答”作为体验核心。

### 1.2 nuwa-skill 的关键差异

nuwa 强调：不是只学“说话风格”，而是抽取**认知框架**（怎么想、怎么判断、什么不做、边界在哪里）。

可借鉴能力：
- Expert 蒸馏时的“五层抽象”：表达、思维模型、决策启发式、反模式、诚实边界。
- 明确“做不到什么”，避免把系统包装成万能真理机。

### 1.3 对你产品的启发总结

你最终应做成一个**双引擎系统**：
- 引擎 A：`Self Distillation Engine`（我的工作方式）
- 引擎 B：`Expert Distillation Engine`（某类专家视角）

再通过一个统一的“任务编排层”，把“我的习惯 + 专家视角 + 企业知识库”合成最终回答或文档产物。

## 2. 产品定位与商业模式

## 2.1 目标用户

- 个人创作者 / 自由职业者：希望有“第二大脑 + 分身”。
- 小团队管理者：需要统一回复口径、知识沉淀、文档自动化。
- 咨询/销售/运营等高频文本工作岗位：重视效率与风格一致性。

## 2.2 核心卖点（你首页就该写这三条）

1. **像你一样工作**：学习你的习惯、表达、优先级。
2. **像专家一样思考**：可切换 Expert 模型辅助决策。
3. **直接产出可交付文件**：一键生成并下载 PPT/Excel/Word/PDF。

## 2.3 订阅分层（建议）

- Free（体验）
  - 1 个工作体
  - 200MB 存储
  - 月度 token 上限
- Pro（个人主力）
  - 3 个工作体（Self + 2 Expert）
  - 20GB 存储
  - 文档生成与导出
- Team（团队）
  - 多成员、共享知识库
  - 权限与审计日志
  - API / Webhook
- Enterprise（私有部署可选）
  - 独立 VPC / 本地化模型
  - 合规与定制策略

## 4. 系统能力设计（核心）

## 4.1 数据层

支持数据类型：
- 文本：txt/md/doc/docx/pdf
- 表格：csv/xlsx
- 演示：ppt/pptx
- 图片：png/jpg（可 OCR）
- 聊天：微信/QQ/IM 导出文本（通过导入模板）

能力：
- 全量索引 + 分块（chunk）+ 向量化。
- 原文可追溯（回答必须可回链到源文档）。
- 用户可下载原文件与处理后结构化结果（JSON / markdown 摘要）。

## 4.2 蒸馏层

### A. 蒸馏自己（Self）
输出对象：
- `Work Memory`：项目、偏好、常用流程、常见决策依据。
- `Work Persona`：表达风格、沟通规则、时间管理偏好。

### B. 蒸馏 Expert
输出对象：
- `Expert Lens`：认知框架、判断准则、反模式、边界声明。

### C. 融合策略（最关键）
最终回答由以下权重融合：
- 企业知识事实（最高优先，防幻觉）
- 用户自我习惯（执行风格）
- Expert 视角（分析维度）
- 当前任务约束（格式、长度、受众）

## 4.3 生成层（Agent Workflow）

典型工作流：
1. 用户提问/下任务。
2. 路由器判断：问答 / 方案 / 文档生成。
3. RAG 检索事实证据。
4. Self/Expert 模块给出风格与决策建议。
5. 策略引擎审查（合规、口径、敏感信息）。
6. 输出结果 + 引用来源 + 可下载文件。

## 5. 关于“对外口径”需求的产品化方案（重点）

你举的示例：
> “青云项目预期收入 2000 万，但对外回复要说项目不挣钱。”

这是一个非常敏感、也非常真实的企业场景。建议产品化为**“沟通策略规则”**，而不是“伪造事实”。

### 建议落地方式

在 `Policy Center` 提供两层配置：

1. **事实层（Fact Layer）**
   - 存储内部真实数据（如预算、收入预期）
2. **表达层（Messaging Layer）**
   - 按受众（内部/外部/合作方）生成不同口径模板

系统输出时要求：
- 标记“内部版/外部版”
- 记录规则来源和变更人
- 保留审计日志，避免法律与合规风险

> 这样既满足业务“口径管理”，又避免把系统设计成“自动造假引擎”。

## 6. 技术架构（部署到你的服务器）

## 6.1 建议技术栈（兼容 Web 快速开发）

- 前端：Next.js + TypeScript + Tailwind
- 后端：Node.js（NestJS 或 Next API）
- 任务队列：Redis + BullMQ
- 数据库：PostgreSQL
- 向量库：pgvector（早期）/ Milvus（中后期）
- 对象存储：S3 兼容（MinIO / AWS S3）
- 文档转换：LibreOffice headless + python-docx + pptx + pandas
- 模型层：
  - 通用大模型（对话/推理）
  - Embedding 模型（检索）
  - 可插拔模型网关
- 部署：Docker Compose（MVP）→ Kubernetes（增长期）

## 6.2 服务拆分

- `web-app`：前端与用户会话
- `api-gateway`：鉴权、限流、计费
- `ingestion-service`：文件解析、清洗、索引
- `distill-service`：Self/Expert 蒸馏
- `rag-service`：检索与引用拼装
- `doc-service`：PPT/Excel/Word/PDF 生成
- `policy-service`：规则引擎与审计
- `billing-service`：订阅、支付、配额

## 6.3 安全与权限

- 每个 Workspace 数据逻辑隔离（Tenant ID）。
- 文件加密存储，下载鉴权与水印（可选）。
- 操作审计日志（上传、下载、导出、策略变更）。
- 默认保留期策略 + 一键导出/删除（满足数据权利）。

## 7. 两版本产品蓝图

## 7.1 版本 A：蒸馏自己（Self Pro）

适用：个人/中小团队

功能包：
- 上传个人工作数据并构建“工作分身”
- 日常问答、任务拆解、工作复盘
- 常规文档生成
- 个人风格回复

KPI：
- 7 日留存
- 每周活跃任务数
- 用户“风格匹配满意度”

## 7.2 版本 B：蒸馏 Expert（Expert Studio）

适用：希望“多专家视角”决策的用户

功能包：
- 自定义创建 Expert（行业顾问、讲师、管理者等）
- 每个 Expert 具有可解释的认知框架
- 可与 Self 组合输出（我 + 专家）

KPI：
- Expert 调用率
- 多视角任务完成率
- 决策建议采纳率（用户打标）

## 9. 风险与对策

1. **幻觉与错误引用**
   - 对策：强制引用 + 无依据不回答
2. **隐私与数据泄露**
   - 对策：加密、最小权限、审计与告警
3. **“口径管理”越界为不当陈述**
   - 对策：策略分级、审批、日志、风险提示
4. **文档生成质量不稳定**
   - 对策：模板化 + 后处理校验
5. **成本失控（模型调用/存储）**
   - 对策：分层套餐 + 缓存 + 队列削峰
